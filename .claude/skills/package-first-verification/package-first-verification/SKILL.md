# Skill: Package-First Verification

## Purpose
Enforce the package-first architecture by ensuring every implementation begins with a scan of **two YAML sources**:

| File | Scope | Maintained by |
| :--- | :--- | :--- |
| `delphi-ai/config/ecosystem_packages.yaml` | **Ecosystem (Global)** packages published for cross-project reuse | Manually in PACED |
| `foundation_documentation/local_packages.yaml` | **Local (Project-Bound)** packages in `packages/` dirs | Auto-generated by `verify_package_registry.sh` |

## Package Tier Model

| Tier | Autonomy | Source of Truth | Agent Behavior |
| :--- | :--- | :--- | :--- |
| **Local** | Total | `local_packages.yaml` | Treat as modular code. Modify freely. Breaking changes OK — fix callers in the same PR. |
| **Ecosystem** | High | `ecosystem_packages.yaml` | Can modify, but version and evaluate cross-project impact. Prefer additive changes. |
| **External** | Low | Stack manifests | Do not modify. Wrap in adapter if behavior needs to change. |

## When to Invoke
- Before planning implementation of any new feature.
- Before creating a new controller, service, repository, utility, or helper.
- Before adding a third-party dependency.
- When a TODO involves creating new files in `app/Services/`, `app/Helpers/`, `lib/utils/`, or similar host-level utility paths.

## Procedure

### 1. Read the YAMLs
Read both files:
```bash
cat delphi-ai/config/ecosystem_packages.yaml
cat foundation_documentation/local_packages.yaml
```

If `local_packages.yaml` does not exist or is stale, run:
```bash
bash delphi-ai/tools/verify_package_registry.sh --project-root <path>
```

### 2. Interpret the Fields

**Ecosystem YAML** — every entry is a published proprietary package. Check the project manifest to see if it is already a dependency.

**Local YAML** — each entry has:

| Field | Meaning |
| :--- | :--- |
| `in_use: true` | Package is a declared dependency — **use it directly** |
| `in_use: false` | Package exists but is NOT a dependency — **recommend adoption** |
| `has_readme: false` | Package is missing documentation — flag as warning |

### 3. Read the README
For each relevant package, read its `README.md` to understand purpose, public API, and integration. The README is the authoritative documentation — not the YAML.

### 4. Evaluate and Decide (Tier-Aware)

| Situation | Local Package | Ecosystem Package | External Package |
| :--- | :--- | :--- | :--- |
| Package covers the need | **Use directly.** Modify API if needed. | **Use directly.** Prefer additive extension. | **Use directly.** Do not modify. |
| Partial match | **Extend.** Breaking changes OK. | **Extend** with additive API. Version bump. | **Wrap** in a local adapter. |
| No match, code is reusable | Create **new local package**. | Propose ecosystem package if cross-project. | N/A |
| Code is host-specific | Implement locally. Document rationale. | N/A | N/A |

### 5. Record the Assessment
Add to the TODO:

```markdown
## Package-First Assessment
- YAMLs consulted: ecosystem_packages.yaml, local_packages.yaml
- Relevant packages found:
  - [Local] <name> — <action taken>
  - [Ecosystem] <name> — <action taken>
  - [External] <name> — <action taken>
- READMEs read: <list>
- Decision: Use <package> / Adopt <package> / Extend <package> / New package <name> / Local implementation
- Tier: Local / Ecosystem / External
- Rationale: <brief>
```

### 6. Post-Implementation
If a new proprietary package was created:
- [ ] Package has `README.md` following canonical format (`delphi-ai/templates/package_readme_template.md`)
- [ ] Run `bash delphi-ai/tools/verify_package_registry.sh` to update `local_packages.yaml`
- [ ] If ecosystem-level, add entry to `delphi-ai/config/ecosystem_packages.yaml`

## Validation
- Package-First Assessment is present in the TODO with tier classification.
- If a proprietary package was found (used or available), the implementation leverages it.
- No new host-level utility files duplicate proprietary package capabilities.
- Local packages are modified directly when needed — no workarounds in host app to avoid touching them.
- External packages are never forked without explicit user approval.
- Both YAMLs are current (re-run script if packages were added/removed).
