## How This Works

1. The script scans `packages/` directories in each stack for **local proprietary packages**.
2. It scans stack manifests (`pubspec.yaml` / `composer.json`) for **ecosystem packages** (non-path dependencies matching the proprietary vendor prefix).
3. Checked `[x]` = package is declared as a dependency (in use).
4. Unchecked `[ ]` = package exists but is NOT declared as a dependency (candidate for adoption).
5. Each entry links to the package README for the agent to read when evaluating relevance.

## Package-First Rule

Before implementing any feature, endpoint, domain, screen, controller, service, or repository:

1. Read this checklist — **both ecosystem and local sections**.
2. For each checked `[x]` package, read its README to evaluate if it covers the planned work.
3. If a match exists, extend the package. Do not duplicate in the host app.
4. For unchecked `[ ]` packages, evaluate if adopting them would avoid new implementation.
5. If no match exists but the work is reusable, consider creating a new proprietary package.
6. Record the assessment in the TODO.

## Promotion Path (Local → Global)

When a local package matures and becomes domain-agnostic, it can be promoted to an ecosystem package.
See `paced.core.package-first` rule for promotion criteria and procedure.
